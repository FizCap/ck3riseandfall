# CK3 GUI Modding Instructions for Rise and Fall

Purpose
- Help AI agents make correct, maintainable CK3 GUI changes for this mod.
- Prioritize verified syntax from local `docs/data_types*.txt` and proven vanilla patterns.
- Reduce common GUI failure loops: wrong scope, wrong datacontext, empty datamodel, stale override, missing loc.

File Placement
- GUI overrides/custom GUI: `gui/`
- Reusable shared tooltip/widget types: `gui/shared/`
- Scripted GUI wrappers: `common/scripted_guis/`
- UI localization: `localization/english/`
- Never edit installed vanilla `game/` files.

Encoding and Style
- Use UTF-8 with BOM for CK3 content files (`.gui`, script `.txt`, localization `.yml`).
- Keep edits ASCII unless the target file already intentionally uses non-ASCII.
- Preserve vanilla style and naming when overriding vanilla windows.
- Keep comments short and only for non-obvious logic.

Runtime Mental Model
- GUI is declarative and expression-driven.
- Game-state mutations should usually go through:
  - command objects (`IsValidCommand` + `CreateCommandPopup`)
  - or scripted GUI wrappers (`ScriptedGui.Execute` -> scripted effect).
- `datacontext` and `datamodel` determine everything:
  - If context is wrong, valid syntax can still do nothing.
  - If model is empty, widgets may "work" but never show.

Vanilla-Proven Patterns To Copy

1) Scripted GUI button with proper validity and tooltip
```gui
button_standard = {
    datacontext = "[GetScriptedGui( 'create_head_of_faith' )]"
    enabled = "[ScriptedGui.IsValid( GuiScope.SetRoot( GetPlayer.MakeScope ).AddScope( 'faith', Faith.MakeScope ).End )]"
    onclick = "[ScriptedGui.Execute( GuiScope.SetRoot( GetPlayer.MakeScope ).AddScope( 'faith', Faith.MakeScope ).End )]"
    tooltip = "[ScriptedGui.BuildTooltip( GuiScope.SetRoot( GetPlayer.MakeScope ).AddScope( 'faith', Faith.MakeScope ).End )]"
}
```

2) Command-object button (safer for vanilla actions)
```gui
button_round = {
    enabled = "[IsValidCommand( TitleViewWindow.GetDestroyTitle )]"
    onclick = "[CreateCommandPopup( TitleViewWindow.GetDestroyTitle )]"
    tooltip = "[TitleViewWindow.GetDestroyTitleButtonTooltip]"
}
```

3) Datamodel guard for empty/non-empty views
```gui
visible = "[DataModelHasItems( TitleViewWindow.GetVassalGroupItems )]"
visible = "[IsDataModelEmpty( TitleViewWindow.GetVassalGroupItems )]"
```

4) Chained datacontext is valid and common
```gui
datacontext = "[TitleViewWindow.GetTitle]"
datacontext = "[Title.GetHolder]"
```
Use this when child context depends on a previous context hop.

5) `oncreate` setup calls are normal
```gui
oncreate = "[BindFoldOutContext]"
oncreate = "[PdxGuiFoldOut.Unfold]"
```

Scripted GUI Wrapper Rules (`common/scripted_guis`)
- Treat scripted GUI entries as script bridges:
  - `scope = <type>`
  - optional `is_shown = {}`
  - optional `is_valid = {}`
  - optional `effect = {}`
- Match wrapper `scope` exactly with the scope passed from GUI.
- Prefer wrapper + scripted effect when a GUI button changes game state.

Recommended wrapper shape
```txt
riseandfall_example_gui = {
    scope = title

    is_shown = { }
    is_valid = { }

    effect = {
        riseandfall_some_effect = yes
    }
}
```

Datamodel and List Rules
- A list UI needs a real data source:
  - hardcoded controller outputs (`...Window.Get...`)
  - or script-filled variable lists via `Scope.GetList( 'name' )`.
- Do not assume a new `.gui` file can invent a controller list.
- If item context is ambiguous, set it explicitly in `item` (example: `datacontext = "[Scope.Char]"` when appropriate).
- Guard list UI with `DataModelHasItems` / `IsDataModelEmpty` to avoid blank layouts.

Text and Localization Rules
- Player-facing text should use localization keys.
- `raw_text` / `raw_tooltip` are acceptable for debug/dev-only labels, but avoid for shipping UX.
- Always add/check loc for:
  - `text = "KEY"`
  - `tooltip = "KEY"`
  - any scripted tooltip path returning keys/descriptions.

Override and Rebase Rules
- When overriding vanilla files by path, keep diffs focused and clearly marked.
- Recheck overridden files after each CK3 patch; full-window copies drift fast.
- Prefer small inserts near existing vanilla blocks over broad rearrangement.
- Use `ignoreinvisible = yes` in row/flow containers when hidden children should not leave gaps.

Validation Workflow
1. Verify syntax token exists:
   - `rg -n "TokenName" docs/data_types*.txt`
2. Verify vanilla usage exists:
   - `rg -n "TokenName" "C:\Program Files (x86)\Steam\steamapps\common\Crusader Kings III\game\gui"`
3. Check scope chain and context:
   - wrapper `scope =` vs passed `MakeScope`
   - `datacontext` hops are intentional and in order
4. Guard lists and command buttons:
   - `DataModelHasItems`/`IsDataModelEmpty`
   - `IsValidCommand` before `CreateCommandPopup`
5. Check localization coverage.
6. Launch CK3 and test exactly the changed screens.
7. If available, use GUI editor/log tools (`dump_data_types` confirms type dump support).

Common Failure Modes (And Fast Fixes)
- Widget never appears:
  - Check `visible` expression first.
  - Check datacontext object validity.
- Button visible but does nothing:
  - If command-based: validate `IsValidCommand(...)`.
  - If scripted GUI-based: verify wrapper scope and `effect` path.
- Tooltip missing or wrong:
  - For scripted GUI, use `BuildTooltip(...)`.
  - Ensure loc keys exist.
- List area blank:
  - Confirm datamodel actually has items.
  - Confirm item context type matches UI calls.
- Override seems ignored:
  - Wrong file path or stale override against newer vanilla.
  - Wrong `blockoverride` name.

Minimal LLM Change Contract
- Inputs required:
  - exact target file/screen
  - intended scope/context
  - any new scripted_gui key(s)
  - any new loc key(s)
- Output required:
  - changed GUI/scripted_gui files
  - loc additions
  - quick smoke-test steps
- Success criteria:
  - parses cleanly
  - visibility/enabled logic matches intent
  - action executes expected command/effect
  - no raw missing loc keys in UI
